# Docker Deployment & Architecture Guide

This document outlines the production-grade Docker architecture designed for the Voice/Audio Verification API. The system has been specifically hardened to integrate flawlessly with a Node.js backend while preventing common deep-learning deployment pitfalls.

## 1. The "Baked" Image Architecture

The traditional approach to deploying AI microservices is to have the Docker container download model weights at runtime. However, downloading 162 MB PANNS weights and various Whisper/ECAPA models every time a container restarts causes severe auto-scaling lag and network vulnerability.

**Our Solution:** The `Dockerfile` has been optimized to "bake" all AI models directly into the image layer during the `docker build` phase.

```dockerfile
# Dockerfile snippet
RUN python -c "from app.verification.models_loader import init_worker_models; init_worker_models()"
```

When DevOps runs `docker build`, this single line of Python is executed. It forces the system to download and cache the following directly to the image's `/root/` directory:
- 162 MB PANNS `.pth` weights from Zenodo.
- `snakers4/silero-vad` from PyTorch Hub.
- `faster-whisper` and `speechbrain/spkrec-ecapa-voxceleb` from HuggingFace.

**Result:** When deployed to production, the container requires absolutely **zero internet access** and boots instantly.

## 2. Preventing C-Level CPU Thrashing

Python's `concurrent.futures.ProcessPoolExecutor` isolates memory between workers, but underlying C-libraries like `OpenBLAS`, `MKL`, and `OMP` (used heavily by `numpy` and `scipy`) completely ignore Python's worker threading limitations. 

If two FastAPI workers process audio simultaneously on a 16-core server, OpenBLAS will attempt to spawn 16 threads per worker (32 threads total), resulting in extreme "CPU Context Switching / Thrashing" that can slow inference down from 1 second to 15 seconds.

**Our Solution:** The `Dockerfile` explicitly disables C-level threading by setting the following environment variables:
```dockerfile
ENV OMP_NUM_THREADS=1
ENV OPENBLAS_NUM_THREADS=1
ENV MKL_NUM_THREADS=1
```
This guarantees that each Python worker operates strictly on a single thread, allowing linear, highly predictable scaling under heavy traffic.

## 3. Node.js Reverse Proxy Integration

This Python API is explicitly designed as a "dumb" heavy-lifting microservice. It relies on a **Node.js Reverse Proxy / API Gateway** to sit in front of it and handle the following critical networking logic:

1. **The "Zip Bomb" Protection:** 
   - **Risk:** A malicious user uploads a 4 GB `.wav` file to crash the Python server via an Out-Of-Memory (OOM) error before the 30-second duration check can execute.
   - **Mitigation:** The Node.js server strictly checks the `Content-Length` of the multipart payload (e.g., max 10 MB) *before* forwarding the request to the Python API.
2. **Queueing & Request Routing:**
   - **Risk:** If the Python API receives 1,000 concurrent requests, the internal Process Pool will be overwhelmed and memory will explode.
   - **Mitigation:** Node.js handles all rate-limiting, authentication, and HTTP request queueing. If the Python API throws a `500 Internal Server Error` (e.g., due to a `BrokenProcessPool` from a corrupted audio segfault), Node.js elegantly handles the retry logic or routes traffic to a healthy Python container replica.

## 4. Deterministic Dependencies

To ensure cross-platform stability and repeatable builds, the Dockerfile relies on a strictly compiled `requirements.txt` generated by `uv`.

```dockerfile
COPY pyproject.toml requirements.txt ./
RUN uv pip install --system -r requirements.txt
```
This guarantees that the exact same library versions are installed every time the container is built, eliminating the risk of a third-party library releasing a breaking update that unexpectedly crashes your production servers.
